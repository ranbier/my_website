Infrastructure

알고리즘보다 먼저 시스템을 멈추게 하는
기반 문제들

실제 로보틱스 시스템에서는 ROS2 설정, MAVROS 브리지, Docker 환경, 네트워크 상태가 전체 생산성과 안정성을 크게 좌우합니다. 이 페이지는 자주 부딪히는 인프라 문제를 구현 관점에서 정리합니다.

ROS2

좋은 노드 구조와 상태 관리가 디버깅 가능성을 만듭니다.

ROS2는 node 기반 구조와 DDS 통신을 제공하지만, 이것만으로 시스템이 자동으로 깔끔해지지는 않습니다. 실제로는 노드 경계, 토픽 구성, lifecycle 상태 전환을 어떻게 설계하느냐가 더 중요합니다.

  • unconfigured -> inactive -> active 흐름을 명확히 관리합니다.
  • 토픽과 네임스페이스를 일관되게 유지해야 디버깅이 쉬워집니다.
  • 시스템이 커질수록 노드 책임 분리가 더 중요해집니다.

MAVROS / MAVLink

FCU와 상위 소프트웨어 스택 사이를 잇는 브리지

MAVROS는 ROS와 Pixhawk 같은 FCU를 연결하는 핵심 레이어입니다. 센서 입력은 FCU에서 자세 안정화와 저수준 제어에 활용되고, MAVLink를 통해 상위 소프트웨어로 전달됩니다. 이 데이터 흐름을 정확히 이해해야 어디에서 문제가 생겼는지 찾기 쉽습니다. 

Sensor -> FCU -> MAVLink -> MAVROS -> ROS

Docker

환경 안정화는 개발 속도와 직접 연결됩니다.

Docker에서는 GUI가 멈추거나 CPU가 과도하게 오르는 문제가 자주 발생합니다. 많은 경우 메모리 부족이나 X11, DISPLAY 설정 문제가 원인입니다. 알고리즘 이슈처럼 보여도 실제로는 개발 환경 문제가 병목이 되는 경우가 많습니다.

  • `free -h`로 메모리 상태를 먼저 확인합니다.
  • DISPLAY와 X11 포워딩 설정을 점검합니다.
  • 컨테이너 안팎의 권한과 마운트 상태를 같이 봅니다.

Network

네트워크 문제는 복잡해 보여도 점검 순서는 단순해야 합니다.

SSH가 안 되거나 장비가 안 보이는 문제를 만났을 때는 큰 가설을 세우기보다 데이터가 실제로 흐를 수 있는 경로부터 차근차근 확인하는 편이 훨씬 빠릅니다.

  1. ping으로 기본 연결 여부 확인
  2. 서브넷과 IP 대역 확인
  3. 케이블과 물리 연결 점검
핵심 인사이트

많은 로봇 프로젝트가 알고리즘에서 막히는 것처럼 보이지만, 실제로는 인프라의 불안정성이 개발과 실험 반복 속도를 훨씬 더 크게 떨어뜨립니다.
